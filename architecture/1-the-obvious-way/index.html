Motivation

The obvious way everyone try to follow

All architectures aim to separate things into three major layers: App, Domain and Data. Other layers can be present in architecture like VIPER for iOS, but those intermediate layer will fall under one of these main three anyway. To recap what each layer is

  • App - is the entry point in the app, it contain the main function that system uses to start you app, the navigation through the app's screens, entry point of the system events, UI elements with response to user stimuli.
  • Domain - code that gathers the required data to be displayed, keeps track of the app's state and updates it in response to events from outside world. The outside world to the domain will be App, and external modules that connect to data base, API or other services.
  • Data - facades, gateways and that code that access the data directly from the API, data base or other services.

That separation is found in any project, be it very well organized or not so well. But because just telling which layer should do what doesn't tell how to write code or how to do the separation in code, it leaves programmers room for many variations of implementations, community is following a template which contains a little bit more details about implementation. In iOS world the templates would be MVC, MVVM, VIPER, Composable, etc. Others would call them architectures, but I do not.

Templates provides details for just one side of your app's architecture. Actually, chosing the template is part of your app's architure. Len Bass is defining the architecture "The software architecture of a program or computing system is the structure or structures of the system that comprise software elements, the externally visible properties of those elements, and the relationship among them"
